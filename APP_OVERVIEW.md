# 端艇部 総合管理アプリ — 詳細概要

## 1. プロジェクト概要

### 1.1 アプリケーション名
**端艇部 総合管理** — 慶應義塾端艇部 ボート部門 専用Webアプリケーション

### 1.2 目的・背景
慶應義塾大学のボート部（端艇部）の日々の練習管理・備品管理・データ分析を一元化するために個人開発したフルスタックWebアプリケーション。従来は紙やLINE、スプレッドシートで分散管理していた情報を、スマートフォンから即座にアクセス・記録できるモバイルファーストのPWA（Progressive Web App）として構築した。

### 1.3 開発規模
| ファイル | 行数 | バイト数 | 内容 |
|---|---|---|---|
| `app.js` | **15,860行** | 673KB | メインロジック（535関数） |
| `supabase-config.js` | **1,387行** | 48KB | バックエンド連携（77関数） |
| `index.html` | **2,138行** | 124KB | UI構造 |
| `styles.css` | **8,524行** | 167KB | スタイリング |
| **合計** | **約28,000行** | **約1MB** | — |

### 1.4 本番URL
Vercelにデプロイし、チームメンバーが日常的に使用中。認証付きのため、部員のみアクセス可能。

## 2. 技術スタック

### 2.1 フロントエンド
- **HTML5 / CSS3 / Vanilla JavaScript**（フレームワーク不使用）
- **PWA対応**：Service Worker (`sw.js`)、`manifest.json` によるオフライン対応・ホーム画面追加
- **レスポンシブデザイン**：モバイルファースト設計、ダークテーマUI
- **Google Fonts (Inter)**：モダンなUI表現
- **CSS変数によるテーマシステム**：一貫したデザイントークン管理

### 2.2 バックエンド（BaaS: Backend as a Service）
- **Supabase**
  - **PostgreSQL データベース**：全データの永続化
  - **Supabase Auth**：メールアドレス/パスワード認証（新規登録・ログイン・セッション管理）
  - **Supabase Storage**：ファイル（画像・ドキュメント）アップロード
  - **Row Level Security (RLS)**：テーブルごとのアクセス制御ポリシー
  - **Edge Functions**：サーバーレス関数（Concept2 API連携のCORS対策）

### 2.3 外部API連携
- **Concept2 LogBook API**（OAuth 2.0）
  - エルゴメーター（ローイングマシン）のワークアウトデータを自動取得
  - 全ページネーション対応で過去データを完全取得
  - トークンリフレッシュによる自動再認証
- **Open-Meteo API**（JMAモデル）
  - 練習時の風速データを自動取得（戸田公園の座標指定）
  - 練習ノートに風況を自動記録

### 2.4 デプロイ・インフラ
- **Vercel**：自動デプロイ（GitHub連携）
- **APIプロキシ**：Vercel Serverless Functions経由でConcept2 APIにアクセス（CORSエラー対策）
- **GitHub**：バージョン管理

### 2.5 データ同期アーキテクチャ
- **ローカルファースト設計**：`localStorage` で即座にデータ保存・表示
- **バックグラウンド同期**：ローカル保存後に非同期でSupabaseに同期
- **初期同期**：ログイン時にSupabaseから全データをローカルにマージ
- **競合解決**：タイムスタンプベースの最新データ優先マージ
- **オフライン対応**：ネットワーク不通時もローカルで動作し、復帰時に同期

## 3. 機能一覧

### 3.1 認証・ユーザー管理

#### メールアドレス認証
- 新規登録（メール・パスワード・名前・ロール選択）
- ログイン / ログアウト
- **管理者承認フロー**：新規ユーザーは管理者の承認後にアクセス可能（承認待ち画面を表示）
- **デモモード**：`?demo=true`パラメータで部外者がテスト可能（Supabaseに保存しない）

#### ロールベースアクセス制御（RBAC）
以下の5ロールに応じてUIと機能を出し分け：
| ロール | 権限概要 |
|---|---|
| **主将** | 全機能アクセス、マスタ管理、ユーザー管理、全体スケジュール編集 |
| **Cox** | 全体閲覧（概況表示）、配艇リギング閲覧、クルーノート |
| **漕手（Rower）** | 自身の練習登録・記録管理、エルゴデータ、練習ノート |
| **コーチ** | 全体閲覧（概況表示）、全員のエルゴデータ閲覧、リギング閲覧 |
| **データ班** | マスタ管理、全体スケジュール編集 |

#### ユーザー管理（管理者機能）
- ユーザー一覧表示
- ユーザー削除機能
- ロール変更
- 承認/拒否

### 3.2 練習スケジュール管理（メイン機能）

#### 週間カレンダー
- **見開き1週間のカレンダー表示**：日〜土の7日間
- **午前/午後+ 追加時間スロット**対応（朝練・午後練・追加練）
- スケジュールの登録・編集・削除
- **前週/次週ナビゲーション**

#### 登録可能なスケジュール種別（9種類）
| 種別 | 詳細 |
|---|---|
| **エルゴ** | エルゴメーターのワークアウト記録（距離・タイム・スプリット・ワット自動計算） |
| **乗艇** | 艇の種別・船名・オール・クルー編成を記録、配艇連動 |
| **ウェイト** | ウエイトトレーニングの種目・重量・セット数 |
| **ラン** | ランニング練習 |
| **参加不可** | 欠席理由を記入 |
| **炊事** | 食事当番（朝・昼・晩の選択） |
| **ビデオ** | ビデオ撮影担当 |
| **伴チャ** | 伴走チャリ担当 |
| **OFF** | 休日 |

#### エルゴ記録の詳細
- 複数セット入力対応
- 距離(m)・タイム入力
- **スプリット自動計算**（500mあたりのペース）
- **ワット自動計算**（Concept2公式: P = 2.80 / pace³）
- **IDT（Ideal Time）計算**：体重を考慮した標準化タイム
  - メニュー別の独自IDT計算式を実装

#### 乗艇記録の詳細
- 艇種選択（8+, 4+, 4-, 4x, 2-, 2x, 1x）
- **配艇からの自動取得**：配艇データと連動し、船・オール・クルーを自動入力
- **シート割り当て**：Cox, Stroke, No.7〜No.2, Bow の各ポジションにメンバーを割り当て
- **クルー自動登録**：1人が登録すると同じクルーの全メンバーに自動でスケジュールを反映

## 7. 開発上のチャレンジと工夫

### 7.1 技術的チャレンジ
- **フレームワーク不使用で28,000行のSPA構築**：仮想DOMやコンポーネントシステムなしで、大規模なシングルページアプリケーションを設計・実装
- **Concept2 OAuth 2.0連携**：CORS制約のある外部APIとの連携を、Vercel Serverless FunctionsとSupabase Edge Functionsで解決
- **データ同期の信頼性**：ローカルファーストアーキテクチャで、オフライン動作とリアルタイム同期を両立
- **重複防止ロジック**：concept2Id → rawId → id の3段階マッチング戦略でデータ重複を完全防止

### 7.2 プロダクト面の工夫
- **チーム全体の練習を可視化**：個人の記録だけでなく、チーム全体の活動状況を一目で把握できるダッシュボード
- **シート管理の自動化**：1人がクルー編成を登録すると、全メンバーのスケジュールに自動反映（ドメイン知識に基づく自動化）
- **風速の自動記録**：気象APIから練習時の風況を自動取得し、練習条件の記録を効率化
- **IDT計算**：体重を考慮したエルゴメータータイムの標準化で、公平なパフォーマンス比較を実現
- **掃除場所ベースの備品管理**：艇庫の実際の運用に即したカテゴリ設計

### 7.3 運用面
- **実際のチームで日常的に使用されている**プロダクション環境

## 9. 使用している主要な設計パターン・技法

- **SPA（Single Page Application）**：ページ遷移なしのタブ切り替え
- **ローカルファーストアーキテクチャ**：オフライン動作対応
- **BaaS（Backend as a Service）パターン**：Supabaseによるサーバーレスバックエンド
- **OAuth 2.0**：外部サービス連携
- **PWA（Progressive Web App）**：ネイティブアプリ風UX
- **RBAC（Role-Based Access Control）**：ロール別機能制御
- **状態管理パターン**：グローバルstateオブジェクトによる一元管理
- **監査ログパターン**：全変更操作の追跡可能性確保
- **イベントドリブンUI**：DOM操作ベースのUI更新

## 10. まとめ

このアプリケーションは、慶應義塾大学ボート部の実際の運用ニーズに基づいて設計・実装した、**約28,000行**の大規模フルスタックWebアプリケーションである。フレームワークに頼らずVanilla JavaScriptで構築し、Supabase BaaSとの連携、Concept2 OAuth API統合、PWA対応、ロールベースアクセス制御、リアルタイムデータ同期など、実務レベルの技術要素を多数含んでいる。

現在もチーム内で日常的に使用されており、ユーザーフィードバックに基づく継続的な機能改善を行っている。
